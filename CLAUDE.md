# Правила работы с проектом SULTANOVA AI

## Что это за проект

Проект больше не рассматривается как обычный личный лендинг-портфолио.

Текущая модель проекта:

- сайт-платформа выбора digital-решений;
- навигация через задачи клиента;
- объяснение разницы между форматами решений;
- акцент на ясности, доверии и прозрачности.

Главный принцип:

**пользователь не обязан разбираться в технологиях — сайт должен помочь ему понять, что ему подходит.**

---

## Главные рабочие документы

Перед заметными изменениями нужно сверяться с:

- `SITE_STRATEGY_CONTEXT.md` — стратегическая база
- `brief.md` — актуальный бриф новой версии сайта
- `research.md` — прикладное исследование
- `project.md` — краткое описание проекта
- `PLAN.md` — текущие приоритеты
- `assets/ASSET_MANIFEST.md` — список и статус ассетов

---

## Основная логика проекта

Сайт строится вокруг:

- задач клиента;
- выбора правильного формата решения;
- диагностики;
- стартовых сценариев;
- кейсов;
- понятного процесса работы;
- доверия и прозрачности.

Сайт не должен скатываться обратно в:

- “просто портфолио”;
- список услуг без маршрутизации;
- технический язык для клиента;
- визуал ради впечатления без пользы.

---

## Правила по содержанию

- Писать для обычного клиента без технического опыта
- Объяснять решения простым языком
- Сохранять честный и спокойный тон
- Не перегружать сайт терминами
- Делать акцент не на “что я умею”, а на “что подходит клиенту”
- Не обещать лишнего
- Если что-то не подходит всем — так и говорить

---

## Правила по визуалу

Сайт должен быть:

- светлым;
- собранным;
- современным;
- технологичным;
- профессиональным;
- человеческим;
- визуально упакованным.

Избегать:

- кислотного AI-стиля;
- случайных неоновых эффектов;
- дешевых абстрактных заглушек;
- перегруза анимациями;
- визуальных решений, которые выглядят как черновик.

Приоритет:

1. Ясная структура
2. Сильный hero
3. Качественные карточки и кейсы
4. Реальные изображения
5. Цельность интерфейса

---

## Правила по ассетам

- Новые изображения и иконки не хранить в корне проекта
- Все curated-ассеты складывать в `assets/`
- Использовать осмысленные имена файлов
- Если ассет уже подготовлен в `assets/`, подключать именно его
- Если нужного визуала нет, сначала добавить его в `assets/ASSET_MANIFEST.md`

Структура:

- `assets/brand/`
- `assets/cases/`
- `assets/icons/`
- `assets/images/`
- `assets/ui/`

---

## Правила по кейсам и портфолио

Кейсы должны показывать:

- задачу;
- выбранный формат решения;
- логику выбора;
- результат;
- реальный визуальный материал.

Кейсы не должны быть просто галереей картинок или списком технологий.

Документы внутри `portfolio/` должны поддерживать новую модель сайта, а не старую логику “портфолио ради портфолио”.

---

## Правила по коду

- HTML и CSS держать читаемыми и масштабируемыми
- Не усложнять реализацию без реальной пользы
- Временные заглушки допустимы только как промежуточный шаг
- Любая визуальная правка должна усиливать упаковку страницы
- Любая смысловая правка должна усиливать ясность для клиента

---

## Перед крупными изменениями

Перед заметным редизайном или изменением структуры нужно понять:

1. Какой блок меняется
2. Какой смысл он должен передавать
3. Какие ассеты для него нужны
4. Есть ли они уже в `assets/`
5. Нужно ли обновить документы после изменений

---

## Что считать хорошим результатом

Хороший результат в этом проекте — это не просто рабочий код.

Это страница, которая:

- выглядит собранно;
- помогает человеку понять, что ему нужно;
- вызывает доверие;
- не перегружает;
- ощущается как готовый продукт;
- поддерживает общую стратегию сайта.
